## 4. Cargo Feature Flags Missing — ✅ resolved 2026-04-30

`Cargo.toml:21–29` previously listed eight runtime dependencies, all unconditional. Resolved by introducing four features that gate the four heaviest deps:

```toml
[features]
default = ["i18n", "colored-display", "yaml", "serde"]
i18n = ["dep:fluent-templates"]
colored-display = ["dep:colored"]
yaml = ["dep:serde_norway", "serde"]
serde = ["dep:serde"]
```

Notes vs. the originally proposed shape:

- All four features are in `default` so existing downstream code keeps compiling. The audit's proposed `default = ["i18n", "colored-display"]` would have been a 0.7.0-shaped break (Razz disappears without `yaml`, serde derives disappear without `serde`); we deferred that decision.
- `yaml` implies `serde` because `cards_from_yaml_str` deserializes into the serde-derived `BasicCard`.
- `rand` stays unconditional — shuffling is core API, not a "decorator" capability.
- `colored::Color` is leaked through `DeckedBase::colors()`, so that trait method is itself `cfg(feature = "colored-display")`. All 12 deck `colors()` impls follow.
- The `Razz` deck moves behind `yaml` (it loads from YAML at runtime); the `Razz` re-export in `prelude` is gated.

Verified by `cargo build` / `cargo test` against the matrix `default`, `--no-default-features`, each single feature, and `--all-features`. All combinations build and test green; under `--no-default-features` the test count drops from 291→239 (color/i18n/yaml-specific tests are cfg-skipped, not failed). The `--no-default-features --examples` build produces only `bridge`/`poker`/`range` (the examples that don't require gated APIs).

This unlocks §5 (`no_std` is now a tractable next step — only `rand`, `log`, `itertools`, `thiserror` remain unconditional) and resolves the deferred question at `localization.rs:25`.

## 5. No `no_std` / WASM Story — 🟡 partially resolved 2026-04-30

Two distinct concerns were bundled here. The WASM half is now done; `no_std` proper remains.

**WASM (✅ done 2026-04-30):** cardpack now builds cleanly for `wasm32-unknown-unknown` across all feature combinations *and* has runtime test coverage on wasm. Changes:

1. Added `getrandom = { version = "0.3", features = ["wasm_js"] }` as a `cfg(target_arch = "wasm32")` dep so consumers automatically get the `wasm_js` feature flag flipped on (browser-targeting backend for `rand_core`).
2. Added `.cargo/config.toml` setting `--cfg getrandom_backend="wasm_js"` for the wasm target so this repo's own builds work; consumers do the same in their own config. Also sets `runner = "wasm-bindgen-test-runner"` so `cargo test --target wasm32-unknown-unknown` dispatches automatically.
3. New `tests/wasm.rs` (5 cases) runs under real wasm via `wasm-bindgen-test`: shuffle determinism, draw, no-arg shuffle (exercises the getrandom wasm_js backend), `DeckKind::all()`, `cards_from_yaml_str` parser. `wasm-bindgen-test = "0.3"` added as target-gated dev-dep.
4. New CI jobs `wasm-build` (compile-only gate, all-features + no-default-features + example) and `wasm-test` (installs `wasm-bindgen-cli` via `taiki-e/install-action`, runs `cargo test --target wasm32-unknown-unknown --test wasm`).

A working `examples/wasm.rs` demonstrates the wasm-friendly patterns (seeded shuffle, no `cards_from_yaml_file`, no `demo_cards`). `docs/wasm.md` covers the consumer setup, recommended feature combos, runtime gotchas, and how to run the wasm tests locally.

**`no_std` (still open):** The remaining blockers for an alloc-only build are `HashMap`/`HashSet` in the public API on `Pile::map_by_suit`, `Pile::into_hashset`, `Ranged::combos`, `Ranged::map_by_rank` (would need `BTreeMap`/`BTreeSet` — public API change), and `rand::rng()` callers (`shuffle()` / `shuffled()` no-arg). See action #13 in §16.

## 6. No Hand Evaluator — ✅ resolved 2026-04-30

`CKCRevised` (per `src/basic/types/traits.rs`) exposes the *primitives* for Cactus Kev encoding (`get_ckc_number`, `ckc_rank_bits`, `ckc_get_prime`, `ckc_rank_shift8`) — but no evaluator builds on top.

Resolved by adding `examples/poker_eval.rs` (~50 lines) that wires `ckc-rs` to `Pile<Standard52>`: deal a Texas Hold'em board, enumerate the 21 5-card combos per player via `Pile::combos(5)`, score each through `ckc_rs::evaluate::five_cards`, and report each player's best hand by `HandRankName` / `HandRankClass`. This closes the README/reality gap without taking on the scope of a built-in evaluator (competitors `rs-poker` and `poker` already cover that at 50M hands/s — out of scope for this crate).

## 7. Localization Breadth/Depth Gap — [M]

The initial framing ("only 3 of 11 decks have FTL files") was misleading. Inspecting `fluent_deck_key()` on each deck shows that **most decks share `french.ftl` via `FLUENT_KEY_BASE_NAME_FRENCH`**:

| Deck | Deck key | FTL file used |
|---|---|---|
| French, Standard52, Euchre24, Euchre32, Spades, Short, Razz | `french` | `french.ftl` ✅ |
| Skat | `skat` | `skat.ftl` ✅ |
| Tarot | `tarot` | `tarot.ftl` ✅ |
| Pinochle, Canasta (pre-2026-04-29) | `pinochle`, `canasta` | ❌ none — silent fallback |

The Pinochle and Canasta gap was closed in this session (see §2). The remaining i18n gaps are:

- **Language breadth**: now `en-US`, `de`, plus drafts in `fr`, `la`, `tlh` — five locales, three of them needing review before promotion. Drafts mark per-file confidence (see each locale's `README.md`); tarot/skat are tradition-sensitive in fr/la/tlh and their READMEs name the reviewer profile required for promotion. No `es`, `it`, `ja`, etc. yet — adding any is pure translation work using the established directory pattern.
- **`src/localization.rs:295`** — `FluentName::new_with_weight` is `todo!()`. If anything calls it on a `FluentName`, it panics. Tests dodge this with a `WeightedName` helper struct (`localization.rs:393–419`); production callers should not exist, but the `todo!()` is still public API surface.

If i18n is the value prop the README advertises, the language breadth (only 2 locales) is the credibility gap.

**Newly discovered while drafting `fr/`** (now resolved — see §2): `de/tarot.ftl` had two latent bugs:

1. It used `name-rank-tarot-{0..l}` for Major Arcana, but cardpack's en-US schema uses `name-rank-tarot-special-{0..l}`. Every German Major Arcana lookup missed, fell back to en-US, and returned the English name. The German translations were dead code.
2. It contained **no Minor Arcana entries** (`name-rank-tarot-{a,k,q,n,p,t,9..2}`), so all 56 Minor Arcana cards in German also fell back to English.

Both were fixed in this session. The `fr/tarot.ftl` was written against the correct schema from the start.

## 8. No Deck Registry — ✅ resolved 2026-04-30

Decks previously lived as 11 distinct types with no non-generic way to enumerate them. Resolved by adding `DeckKind` (`src/basic/decks/registry.rs`) — a non-generic enum with one variant per shipped deck (`Razz` is `cfg(feature = "yaml")`) and methods `all() -> &'static [DeckKind]`, `deck_name()`, `base_vec() -> Vec<BasicCard>`, `fluent_deck_key()`, and (under default features) `demo(verbose)`. Re-exported from `prelude` so `for kind in DeckKind::all() { kind.demo(true) }` Just Works.

`examples/demo.rs` gained an `--all` flag that uses the registry; the existing per-deck flags remain for backward-compat. The unit tests verify length, deduplication, parity with the typed deck's `base_vec()` / `deck_name()`, and that `Razz` only appears with `yaml`.

## 9. Property-Based Tests Absent — ✅ resolved 2026-04-30

The 94 unit tests cover specific known cases. Card libraries are unusually well-suited to property tests:

```rust
proptest! {
    #[test]
    fn shuffled_deck_reconstructs(seed: u64) {
        let original = Standard52::deck();
        let shuffled = original.clone().shuffled_with_seed(seed);
        prop_assert!(original.same(&shuffled));
    }

    #[test]
    fn pile_on_is_associative(...) { ... }

    #[test]
    fn draw_n_then_extend_is_identity(n in 0..52u8) { ... }
}
```

Note (✅ resolved 2026-04-29): seeded shuffle (`shuffle_with_seed` /
`shuffled_with_seed`) plus generic-RNG variants (`shuffle_with_rng` /
`shuffled_with_rng`) now exist on both `BasicPile` and `Pile<T>`. The
`basic_pile.rs:63` TODO is removed. Property testing (row #4) is
unblocked.

Resolved 2026-04-30 — `tests/properties.rs` adds 10 proptest cases
covering the listed invariants and a few extras: shuffle preserves
multiset, shuffle is deterministic from seed, shuffle preserves
length, sort is idempotent, shuffle+sort is canonical, draw preserves
total count, draw too many returns None and leaves deck untouched,
draw zero is no-op, pile_on sums lengths, pile_on of one is identity.
`proptest = "1"` added to `[dev-dependencies]`.

## 10. No Benchmarks — [L]

No `benches/`, no Criterion. The 2026-04-09 audit acknowledges `draw_first()` is O(n) but defers because decks are small. Without baseline benches, a future regression (e.g., switching `Vec` to `LinkedList`, or growing `Canasta` to multiple shoes) goes undetected. A 30-line `benches/draw.rs` covering `shuffle`, `draw(1)`, `draw(13)`, `pile_on(8 piles)` for the largest deck (Canasta, 108 cards) gives a perf safety net.

## Overall Assessment

The 2026-04-09 audit gave 8.5/10 for *correctness*. By that lens little has changed. By the *completeness and ergonomics* lens this document uses, the largest deltas are:

- **README accuracy and Cargo features** (§3, §4) — both are user-facing and quick to fix.
- **i18n breadth** (§7) — the marketing claim is ahead of the implementation.
- **Test sophistication** (§9, §10, §11) — property tests, benches, and mutation closure are the natural progression after unit tests are stable.

The crate is production-ready for the deck-representation use case. It is not yet a one-stop shop for poker engines, blackjack scorers, or embedded card games — and the gap analysis is mostly about deciding which of those it wants to be.
