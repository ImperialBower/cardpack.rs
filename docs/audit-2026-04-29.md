## 3. Public README Rot — [H]

The README is the front door on crates.io and docs.rs. Several entries are wrong today.

- **`README.md:79–80`** — links to `src/fluent/locales/en-US/french-deck.ftl` and `src/fluent/locales/de/french-deck.ftl`. Actual path is `src/localization/locales/...`. Both links 404 on GitHub.
- **`README.md:204`** — references `serde_yml`. The crate uses `serde_norway` per `Cargo.toml:28`.
- **`README.md:88`** — claims responsibility "Verify that a specific card is playable given a set of discards." No `is_playable` API exists on `Pile<T>` or `Card<T>`. Either implement, soften the claim, or delete the bullet.

## 4. Cargo Feature Flags Missing — [H]

`Cargo.toml:21–29` lists eight runtime dependencies, **all unconditional**: `colored`, `fluent-templates`, `itertools`, `log`, `rand`, `serde`, `serde_norway`, `thiserror`. There are no `[features]`. Implications:

- Users who don't need i18n still pull in `fluent-templates` (~30+ transitive deps).
- Users who don't need YAML deck loading still pull in `serde_norway`.
- Users who don't want serde derives still pay for `serde` derive macros.
- No `default-features = false` story for embedded / WASM consumers.

**Suggested feature taxonomy**:

```toml
[features]
default = ["i18n", "colored-display"]
i18n = ["dep:fluent-templates"]
colored-display = ["dep:colored"]
yaml = ["dep:serde_norway"]
serde = ["dep:serde"]
```

Then gate the relevant modules with `#[cfg(feature = "...")]`. This unlocks Findings §5 (`no_std`) and partially resolves the question deferred in `localization.rs:25`.

## 5. No `no_std` / WASM Story — [M]

The crate compiles only against `std`. Hard blockers: `fluent-templates` (uses filesystem at compile time but runtime allocators), `colored` (TTY detection), `std::collections::HashMap` use in `Pile`, file I/O in `BasicCard::cards_from_yaml_file`. With feature flags (§4), an `alloc`-only core is plausible: `BasicCard`, `BasicPile`, `Pile<T>`, `Pip` are all data + algorithms, no std-specific types required.

---

## 6. No Hand Evaluator — [M]

`CKCRevised` (per `src/basic/types/traits.rs`) exposes the *primitives* for Cactus Kev encoding (`get_ckc_number`, `ckc_rank_bits`, `ckc_get_prime`, `ckc_rank_shift8`) — but no evaluator builds on top to classify a 5- or 7-card hand as "two pair" / "straight" / "flush". The `ckc-rs` crate is a dev-dependency (`Cargo.toml:32`) used in tests but not exposed.

Two options:

- Implement evaluation in cardpack (scope creep — competitors `rs-poker` and `poker` do this at 50M hands/s).
- Document the integration pattern explicitly: add an `examples/poker_eval.rs` showing how to wire `ckc-rs` to `Pile<Standard52>`.

The README already advertises poker examples; the latter (~40 lines of example code) closes the marketing/reality gap cheaply.

## 7. Localization Breadth/Depth Gap — [M]

The initial framing ("only 3 of 11 decks have FTL files") was misleading. Inspecting `fluent_deck_key()` on each deck shows that **most decks share `french.ftl` via `FLUENT_KEY_BASE_NAME_FRENCH`**:

| Deck | Deck key | FTL file used |
|---|---|---|
| French, Standard52, Euchre24, Euchre32, Spades, Short, Razz | `french` | `french.ftl` ✅ |
| Skat | `skat` | `skat.ftl` ✅ |
| Tarot | `tarot` | `tarot.ftl` ✅ |
| Pinochle, Canasta (pre-2026-04-29) | `pinochle`, `canasta` | ❌ none — silent fallback |

The Pinochle and Canasta gap was closed in this session (see §2). The remaining i18n gaps are:

- **Language breadth**: now `en-US`, `de`, plus drafts in `fr`, `la`, `tlh` — five locales, three of them needing review before promotion. Drafts mark per-file confidence (see each locale's `README.md`); tarot/skat are tradition-sensitive in fr/la/tlh and their READMEs name the reviewer profile required for promotion. No `es`, `it`, `ja`, etc. yet — adding any is pure translation work using the established directory pattern.
- **`src/localization.rs:295`** — `FluentName::new_with_weight` is `todo!()`. If anything calls it on a `FluentName`, it panics. Tests dodge this with a `WeightedName` helper struct (`localization.rs:393–419`); production callers should not exist, but the `todo!()` is still public API surface.

If i18n is the value prop the README advertises, the language breadth (only 2 locales) is the credibility gap.

**Newly discovered while drafting `fr/`** (now resolved — see §2): `de/tarot.ftl` had two latent bugs:

1. It used `name-rank-tarot-{0..l}` for Major Arcana, but cardpack's en-US schema uses `name-rank-tarot-special-{0..l}`. Every German Major Arcana lookup missed, fell back to en-US, and returned the English name. The German translations were dead code.
2. It contained **no Minor Arcana entries** (`name-rank-tarot-{a,k,q,n,p,t,9..2}`), so all 56 Minor Arcana cards in German also fell back to English.

Both were fixed in this session. The `fr/tarot.ftl` was written against the correct schema from the start.

## 8. No Deck Registry — [M]

Decks live as 11 distinct types (`French`, `Standard52`, `Pinochle`, …). There is no enum, no `Vec<&'static dyn Decked>`, no `for deck in cardpack::decks()`. Users must know the type names. The `examples/demo.rs` enumerates them via clap flags but the library itself doesn't.

A simple addition: a `DeckKind` enum with variants for each shipped deck and a `DeckKind::deckvec()` method returning the cards. Useful for CLIs, GUIs, and game launchers.

## 9. Property-Based Tests Absent — [M]

The 94 unit tests cover specific known cases. Card libraries are unusually well-suited to property tests:

```rust
proptest! {
    #[test]
    fn shuffled_deck_reconstructs(seed: u64) {
        let original = Standard52::deck();
        let shuffled = original.clone().shuffled_with_seed(seed);
        prop_assert!(original.same(&shuffled));
    }

    #[test]
    fn pile_on_is_associative(...) { ... }

    #[test]
    fn draw_n_then_extend_is_identity(n in 0..52u8) { ... }
}
```

Note: this also exposes a missing API — `shuffled_with_seed` doesn't exist (per `basic_pile.rs:63` TODO "Shuffle seed parameter"). Adding deterministic shuffling unblocks property testing.

## 10. No Benchmarks — [L]

No `benches/`, no Criterion. The 2026-04-09 audit acknowledges `draw_first()` is O(n) but defers because decks are small. Without baseline benches, a future regression (e.g., switching `Vec` to `LinkedList`, or growing `Canasta` to multiple shoes) goes undetected. A 30-line `benches/draw.rs` covering `shuffle`, `draw(1)`, `draw(13)`, `pile_on(8 piles)` for the largest deck (Canasta, 108 cards) gives a perf safety net.

## 12. CI Gaps — [L]

`.github/workflows/CI.yaml` runs tests, clippy, fmt, semver. Missing:

- **Coverage gate**: no `cargo-llvm-cov` / `tarpaulin`. No badge.
- **`cargo doc` warning gate**: broken intra-doc links and missing-docs warnings ship to docs.rs without failing CI.
- **`cargo audit`**: `audit.yml` runs `cargo deny check advisories` (which covers most of it), but `cargo audit` is the canonical RustSec runner; consider both or document the choice.
- **MSRV regression**: 1.85.0 is in the matrix but no separate "MSRV-only" job that fails on accidental MSRV bumps.

## 14. Idiomatic-Rust Integrations — [L]

- **Const constructors**: only ~1 `const fn` in the codebase. `Pip::new`, `BasicCard::new` could be const, enabling compile-time deck literals.
- **`IntoIterator for &Pile<T>`**: inconsistent across types — borrowing a `&Pile` should iterate `&Card<T>` without forcing `.iter()`.
- **Shuffle seed**: `basic_pile.rs:63` TODO. Without it, no determinism for tests, replays, or networked games.

## Overall Assessment

The 2026-04-09 audit gave 8.5/10 for *correctness*. By that lens little has changed. By the *completeness and ergonomics* lens this document uses, the largest deltas are:

- **README accuracy and Cargo features** (§3, §4) — both are user-facing and quick to fix.
- **i18n breadth** (§7) — the marketing claim is ahead of the implementation.
- **Test sophistication** (§9, §10, §11) — property tests, benches, and mutation closure are the natural progression after unit tests are stable.

The crate is production-ready for the deck-representation use case. It is not yet a one-stop shop for poker engines, blackjack scorers, or embedded card games — and the gap analysis is mostly about deciding which of those it wants to be.
