## 4. Cargo Feature Flags Missing ‚Äî ‚úÖ resolved 2026-04-30

`Cargo.toml:21‚Äì29` previously listed eight runtime dependencies, all unconditional. Resolved by introducing four features that gate the four heaviest deps:

```toml
[features]
default = ["i18n", "colored-display", "yaml", "serde"]
i18n = ["dep:fluent-templates"]
colored-display = ["dep:colored"]
yaml = ["dep:serde_norway", "serde"]
serde = ["dep:serde"]
```

Notes vs. the originally proposed shape:

- All four features are in `default` so existing downstream code keeps compiling. The audit's proposed `default = ["i18n", "colored-display"]` would have been a 0.7.0-shaped break (Razz disappears without `yaml`, serde derives disappear without `serde`); we deferred that decision.
- `yaml` implies `serde` because `cards_from_yaml_str` deserializes into the serde-derived `BasicCard`.
- `rand` stays unconditional ‚Äî shuffling is core API, not a "decorator" capability.
- `colored::Color` is leaked through `DeckedBase::colors()`, so that trait method is itself `cfg(feature = "colored-display")`. All 12 deck `colors()` impls follow.
- The `Razz` deck moves behind `yaml` (it loads from YAML at runtime); the `Razz` re-export in `prelude` is gated.

Verified by `cargo build` / `cargo test` against the matrix `default`, `--no-default-features`, each single feature, and `--all-features`. All combinations build and test green; under `--no-default-features` the test count drops from 291‚Üí239 (color/i18n/yaml-specific tests are cfg-skipped, not failed). The `--no-default-features --examples` build produces only `bridge`/`poker`/`range` (the examples that don't require gated APIs).

This unlocks ¬ß5 (`no_std` is now a tractable next step ‚Äî only `rand`, `log`, `itertools`, `thiserror` remain unconditional) and resolves the deferred question at `localization.rs:25`.

## 5. No `no_std` / WASM Story ‚Äî [M]

The crate compiles only against `std`. Hard blockers: `fluent-templates` (uses filesystem at compile time but runtime allocators), `colored` (TTY detection), `std::collections::HashMap` use in `Pile`, file I/O in `BasicCard::cards_from_yaml_file`. With feature flags (¬ß4), an `alloc`-only core is plausible: `BasicCard`, `BasicPile`, `Pile<T>`, `Pip` are all data + algorithms, no std-specific types required.

## 6. No Hand Evaluator ‚Äî ‚úÖ resolved 2026-04-30

`CKCRevised` (per `src/basic/types/traits.rs`) exposes the *primitives* for Cactus Kev encoding (`get_ckc_number`, `ckc_rank_bits`, `ckc_get_prime`, `ckc_rank_shift8`) ‚Äî but no evaluator builds on top.

Resolved by adding `examples/poker_eval.rs` (~50 lines) that wires `ckc-rs` to `Pile<Standard52>`: deal a Texas Hold'em board, enumerate the 21 5-card combos per player via `Pile::combos(5)`, score each through `ckc_rs::evaluate::five_cards`, and report each player's best hand by `HandRankName` / `HandRankClass`. This closes the README/reality gap without taking on the scope of a built-in evaluator (competitors `rs-poker` and `poker` already cover that at 50M hands/s ‚Äî out of scope for this crate).

## 7. Localization Breadth/Depth Gap ‚Äî [M]

The initial framing ("only 3 of 11 decks have FTL files") was misleading. Inspecting `fluent_deck_key()` on each deck shows that **most decks share `french.ftl` via `FLUENT_KEY_BASE_NAME_FRENCH`**:

| Deck | Deck key | FTL file used |
|---|---|---|
| French, Standard52, Euchre24, Euchre32, Spades, Short, Razz | `french` | `french.ftl` ‚úÖ |
| Skat | `skat` | `skat.ftl` ‚úÖ |
| Tarot | `tarot` | `tarot.ftl` ‚úÖ |
| Pinochle, Canasta (pre-2026-04-29) | `pinochle`, `canasta` | ‚ùå none ‚Äî silent fallback |

The Pinochle and Canasta gap was closed in this session (see ¬ß2). The remaining i18n gaps are:

- **Language breadth**: now `en-US`, `de`, plus drafts in `fr`, `la`, `tlh` ‚Äî five locales, three of them needing review before promotion. Drafts mark per-file confidence (see each locale's `README.md`); tarot/skat are tradition-sensitive in fr/la/tlh and their READMEs name the reviewer profile required for promotion. No `es`, `it`, `ja`, etc. yet ‚Äî adding any is pure translation work using the established directory pattern.
- **`src/localization.rs:295`** ‚Äî `FluentName::new_with_weight` is `todo!()`. If anything calls it on a `FluentName`, it panics. Tests dodge this with a `WeightedName` helper struct (`localization.rs:393‚Äì419`); production callers should not exist, but the `todo!()` is still public API surface.

If i18n is the value prop the README advertises, the language breadth (only 2 locales) is the credibility gap.

**Newly discovered while drafting `fr/`** (now resolved ‚Äî see ¬ß2): `de/tarot.ftl` had two latent bugs:

1. It used `name-rank-tarot-{0..l}` for Major Arcana, but cardpack's en-US schema uses `name-rank-tarot-special-{0..l}`. Every German Major Arcana lookup missed, fell back to en-US, and returned the English name. The German translations were dead code.
2. It contained **no Minor Arcana entries** (`name-rank-tarot-{a,k,q,n,p,t,9..2}`), so all 56 Minor Arcana cards in German also fell back to English.

Both were fixed in this session. The `fr/tarot.ftl` was written against the correct schema from the start.

## 8. No Deck Registry ‚Äî [M]

Decks live as 11 distinct types (`French`, `Standard52`, `Pinochle`, ‚Ä¶). There is no enum, no `Vec<&'static dyn Decked>`, no `for deck in cardpack::decks()`. Users must know the type names. The `examples/demo.rs` enumerates them via clap flags but the library itself doesn't.

A simple addition: a `DeckKind` enum with variants for each shipped deck and a `DeckKind::deckvec()` method returning the cards. Useful for CLIs, GUIs, and game launchers.

## 9. Property-Based Tests Absent ‚Äî [M]

The 94 unit tests cover specific known cases. Card libraries are unusually well-suited to property tests:

```rust
proptest! {
    #[test]
    fn shuffled_deck_reconstructs(seed: u64) {
        let original = Standard52::deck();
        let shuffled = original.clone().shuffled_with_seed(seed);
        prop_assert!(original.same(&shuffled));
    }

    #[test]
    fn pile_on_is_associative(...) { ... }

    #[test]
    fn draw_n_then_extend_is_identity(n in 0..52u8) { ... }
}
```

Note (‚úÖ resolved 2026-04-29): seeded shuffle (`shuffle_with_seed` /
`shuffled_with_seed`) plus generic-RNG variants (`shuffle_with_rng` /
`shuffled_with_rng`) now exist on both `BasicPile` and `Pile<T>`. The
`basic_pile.rs:63` TODO is removed. Property testing (row #4) is
unblocked.

## 10. No Benchmarks ‚Äî [L]

No `benches/`, no Criterion. The 2026-04-09 audit acknowledges `draw_first()` is O(n) but defers because decks are small. Without baseline benches, a future regression (e.g., switching `Vec` to `LinkedList`, or growing `Canasta` to multiple shoes) goes undetected. A 30-line `benches/draw.rs` covering `shuffle`, `draw(1)`, `draw(13)`, `pile_on(8 piles)` for the largest deck (Canasta, 108 cards) gives a perf safety net.

## Overall Assessment

The 2026-04-09 audit gave 8.5/10 for *correctness*. By that lens little has changed. By the *completeness and ergonomics* lens this document uses, the largest deltas are:

- **README accuracy and Cargo features** (¬ß3, ¬ß4) ‚Äî both are user-facing and quick to fix.
- **i18n breadth** (¬ß7) ‚Äî the marketing claim is ahead of the implementation.
- **Test sophistication** (¬ß9, ¬ß10, ¬ß11) ‚Äî property tests, benches, and mutation closure are the natural progression after unit tests are stable.

The crate is production-ready for the deck-representation use case. It is not yet a one-stop shop for poker engines, blackjack scorers, or embedded card games ‚Äî and the gap analysis is mostly about deciding which of those it wants to be.
