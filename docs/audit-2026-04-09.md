## 5. High Priority

### Non-deterministic Canasta Test

**Location:** `src/basic/decks/canasta.rs:182–189`

```rust
// BUG: shuffled() produces non-deterministic output
let pile = Canasta::deck().shuffled();
let expected = "H~D~J~S~H~D~C~S~H~D~C";
let ranks_index = pile.suits_index("~");
assert_eq!(ranks_index, expected);  // will fail randomly
```

The test shuffles the deck and then asserts against a fixed string. This is a latent flake — it will pass most runs due to the specific deck composition but will eventually fail non-deterministically. The test comment `// TODO: WTF??!!` / `// TODO: WTF do I mean by WTF??? Don't do this.` acknowledges this.

**Fix:** Remove the shuffle, or assert on a property of the result (e.g., length, sorted output) rather than a fixed string.

### Silent YAML Load Failure in Razz

**Location:** `src/basic/decks/razz.rs:26–27`

```rust
BasicCard::cards_from_yaml_file("src/basic/decks/yaml/razz.yaml")
    .unwrap_or_else(|_| Vec::default())
```

A missing or malformed YAML file silently returns an empty deck. The doc comment on `Razz` literally says: _"In a production system, I would add at least logging in order to have some record of what's going on."_ — but it hasn't been done yet.

**Fix:** Add a `log::error!` call inside the `unwrap_or_else` closure.

## 6. Medium Priority

### `draw_first()` is O(n)

**Location:** `src/basic/types/pile.rs:215`

```rust
_ => Some(self.remove(0)),  // shifts entire Vec on every draw
```

Removing from the front of a `Vec` shifts all remaining elements. The author has noted this at `pile.rs:597` with a TODO to consider `VecDeque`. Since decks are small (≤108 cards) this is not a practical performance problem today, but it's architecturally inconsistent.

The root cause is documented at `basic_card.rs:24`: cards are sorted high-to-low (reversed `Ord`), but `draw()` takes from the front. If the end-of-vector were treated as the top of the deck, `draw()` could use `pop()` (O(1)).

## 7. Low Priority / Design Debt

### `connectors()` Double-Iterates

**Location:** `src/basic/types/combos.rs:73–80`

```rust
self.iter()
    .filter(|pile| pile.is_connector())
    .cloned()
    .collect::<Self>()       // first full pass → allocates
    .iter()
    .map(|pile| pile.clone().sorted_by_rank())
    .collect::<Self>()       // second full pass
```

Both operations can be chained in a single iterator pass. Author has noted `// TODO: RF - This code is ugly AF.`

### Fluent Templates May Be Unnecessary

**Location:** `src/localization.rs:25`

`// TODO: RF is it time to remove fluent templates all together?`

The `fluent-templates` dependency adds non-trivial complexity and build weight. If localization beyond English is not a core requirement, this could be simplified to direct string lookups.

### `Pip::PRIMES` Marked as HACK

**Location:** `src/basic/types/pips.rs:97–102`

```rust
/// TODO: HACK
pub const PRIMES: [usize; 60] = [ 2, 3, 5, 7, 11, ... ];
```

The intended design is to have primes assigned per-pip rather than looked up by index. The array approach works but is fragile if pip count changes.

### `BasicPileCell` Interior Mutability Pattern

**Location:** `src/basic/types/basic.rs`

The `Cell<BasicPile>` wrapper uses a take-mutate-put pattern throughout:
```rust
let inner_pile = self.0.take();
let result = inner_pile.contains(card);
self.0.set(inner_pile);
```
This is unusual and undocumented. Worth either documenting the rationale or refactoring.

### `ckc_suit_number()` Placement

**Location:** `src/basic/types/basic_card.rs:108`

`// TODO: This needs to be moved out of Basic. Maybe a trait? Maybe just move it out altogether?`

CKC-specific logic in the base `BasicCard` type couples the card data model to a poker evaluation concern.

## 10. Architecture Assessment

**Strengths:**
- `Pile<DeckType>` + `PhantomData` — clean zero-cost generics; deck type is purely compile-time
- `DeckedBase` (data) / `Decked` (behavior) separation keeps deck implementations minimal
- Trait-based design (`CKCRevised`, `Ranged`, `Decked`) is well-partitioned
- 11 deck implementations share the same infrastructure with ~40–100 lines of per-deck code
- `#![warn(clippy::unwrap_used, clippy::expect_used)]` with zero clippy output — strict lint gates enforced
- Zero `unsafe` blocks anywhere in the library

**Concerns:**
- Vector ordering: reversed `Ord` + front-of-Vec draw is architecturally inconsistent
- `BasicPileCell` interior mutability is unusual and undocumented
- Fluent templates add complexity that may exceed their value

## Overall Assessment

**Score: 8.5 / 10**

The cardpack.rs library is a well-engineered, feature-rich playing card library. The architecture is sound, the test suite is comprehensive, and the code is idiomatic Rust with strict clippy compliance. The most actionable item is the non-deterministic canasta test. Everything else represents design debt the author is already aware of and tracking via TODO comments.
